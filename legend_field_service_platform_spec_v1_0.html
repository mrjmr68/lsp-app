Lean Owner-Operator UX Direction

The next frontend direction is a five-person owner-operator model: all team members share the same practical operating view, and the interface prioritizes today's work over role-specific dashboards. The product should feel like a field tool, not an admin portal.

One Task Per Screen

Long pages are replaced by focused steps. Transit, diagnosis notes, resolution triage, parts-needed entry, and invoice approval each get their own screen.

Thumb-Friendly

Buttons and inputs use large touch targets, full-width actions, bold typography, and generous spacing for technicians working on phones, in trucks, or with gloves.

Shared Reality

The UI should not hide operational screens behind role-based navigation. Authentication remains required, but the five-person crew operates from the same daily board and workflow surfaces.

Shared Crew Responsibilities

v1.0 no longer treats the UI as separate role-based dashboards. The language below describes common responsibilities, not restricted navigation. All authenticated crew members should be able to reach the same core daily workflow.

Technician

  • Starts from Home or Today's Jobs.
  • Opens assigned visits.
  • Updates en route / on site / complete status.
  • Records access notes and quick notes.
  • Selects repair templates or marks parts needed.
  • Can create a new billable Add Unit request from the active visit.

Dispatcher

  • Creates requests through Quick Add or planning.
  • Assigns visits to technicians.
  • Uses Today's Jobs to monitor the day's work.
  • Uses Quick Update for phone-call and office updates.

Owner / Admin

  • Reviews invoice-ready work.
  • Confirms selected repair lines and pricing context.
  • Applies flat-rate override for variable or unpriced repairs.
  • Approves invoices, creates PDF, stores snapshot, and sends email when configured.
  • Manages catalog/admin surfaces.

Data Model

v1.0 introduces request/visit language without deleting the existing jobs foundation. This is a transition architecture: new screens operate on requests and visits, while legacy screens continue reading and writing jobs until the cutover is complete.

v1.0 Object Table Purpose
Service Request service_requests The billable or operational customer need. A request can have one or more visits.
Service Visit service_visits The scheduled field event assigned to a technician.
Visit Note visit_notes General, access, status, timing, approval, or system notes tied to one visit.
Visit Repair visit_repairs Selected repair template snapshot for a specific visit. Protects history from future catalog edits.
Parts Needed visit_parts_needed Parts-needed hold record. Billing waits for the return visit.
Invoice Snapshot job_invoice_snapshots Durable invoice record now linkable to service request and service visit.
Transition rule: existing jobs records remain readable. New v1.0 flows create or mirror jobs, service_requests, and service_visits so the older planning and invoice screens keep working while the primary UI moves to visits.

Business Rules

Add Unit

Add Unit always creates a new, separately billable service request and first service visit. It copies the customer and location from the origin visit and stores origin_visit_id.

Parts Needed

Parts-needed visits are not billed at the first visit. The current visit is marked blocked_parts_return, needs_return_visit = true, and no_invoice_until_return_complete = true.

Repair Templates

Repair selections snapshot the catalog values into visit_repairs. Later catalog edits must not rewrite historical invoices or field selections.

Accounting Boundary

No QuickBooks sync in v1.0. Invoice-ready means the app has an internal durable record and PDF/email workflow, not an accounting-system export.

Screen Specifications

Home

  • Phone-first operating dashboard.
  • Entry points: Today's Jobs, Add Job, Quick Update, Planning, Invoice Review.
  • Shows invoice-ready and waiting-parts counts where useful.

Today's Jobs

  • Lists active visits for the current date.
  • Displays status, priority, customer, location, unit, tech, billing status, and access state.
  • Opens /visits/[id].

Quick Add Job

  • Creates a service call in under 30 seconds.
  • Supports customer/location/unit context, issue, priority, and assigned tech.
  • Creates legacy job plus request/visit bridge rows.

Quick Update

  • Fast note/status/access updates.
  • Writes visit notes and mirrors important information to legacy job messages/events.

Active Visit

  • Status buttons: scheduled, en route, on site.
  • Notes and access confirmation.
  • Repair template selection.
  • Parts-needed outcome.
  • Complete visit as repair completed or closed no action.
  • Add Unit creates a new billable request.

Invoice Review

  • Owner-only review queue.
  • Shows selected visit repairs if present.
  • Variable or unpriced visit repairs require owner override.
  • Approval generates PDF, stores invoice snapshot, emails when configured, and marks visit invoiced.

Repair Catalog Contract

The original spreadsheet is a one-time seed source. After import, the app is the system of record for repair templates and pricing constants.

Catalog Field Expected Use
Scope and categorization Location, equipment, component, action, category, and repair code support browsing/search and diagnosis consistency.
Customer description Customer-facing invoice text, preferably QB-ready language.
Technician notes Internal work guidance shown to staff where appropriate.
Flat rate and variable flag Defines whether invoice review can approve directly or requires owner pricing override.
Time and cost assumptions Travel time, work time, labor cost, parts/materials, refrigerant, profit, and margin support auditability.

Testing And Launch Gates

Smoke Test

  1. Log in on desktop and mobile.
  2. Create a Quick Add job.
  3. Assign a technician.
  4. Open from Today's Jobs.
  5. Move through en route and on site.
  6. Add access note and quick note.
  7. Select repair template.
  8. Complete visit as invoice-ready.
  9. Review and approve invoice.

Parts Needed Test

  1. Open an active visit.
  2. Mark parts needed with quantity and note.
  3. Confirm visit billing status is blocked.
  4. Confirm no invoice is sent from the first visit.
  5. Create/complete return visit before billing.

Add Unit Test

  1. Open an active visit.
  2. Use Add Unit.
  3. Confirm a new billable request exists.
  4. Confirm it carries the same customer/location.
  5. Confirm it links back to origin visit.
